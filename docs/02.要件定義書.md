## 2. システム目的

入金・支払の予定および実績を管理し、抜け漏れ防止と業務効率化を実現する。

---

## 3. ユーザー定義

| ユーザー種別 | 説明 |
|-------------|------|
| 一般ユーザー | 入出金の登録・閲覧・管理を行う |

※ 管理者機能は初期スコープ外とする。

---

## 4. 機能要件

### 4.1 認証機能

| ID | 機能 | 説明 |
|----|------|------|
| AUTH-001 | サインアップ | メールアドレス・パスワードでアカウント作成 |
| AUTH-002 | ログイン | メールアドレス・パスワードで認証し、JWT トークンを発行 |
| AUTH-003 | ログアウト | リフレッシュトークンを無効化 |
| AUTH-004 | トークンリフレッシュ | リフレッシュトークンでアクセストークンを再発行 |

**ビジネスルール:**
- パスワードは8文字以上、英数字混在必須
- アクセストークン有効期限: 30分
- リフレッシュトークン有効期限: 7日
- メールアドレスはシステム内で一意

### 4.2 入出金ルール管理機能

| ID | 機能 | 説明 |
|----|------|------|
| RULE-001 | ルール登録 | 入出金の繰り返しルールを登録 |
| RULE-002 | ルール一覧取得 | ログインユーザーのルール一覧を取得 |
| RULE-003 | ルール詳細取得 | 指定ルールの詳細を取得 |
| RULE-004 | ルール更新 | ルール内容を更新 |
| RULE-005 | ルール削除 | ルールを論理削除 |

**ビジネスルール:**
- 種別は `income`（入金）または `expense`（支払）
- 繰り返し種別は `once`（単発）または `monthly`（毎月）
- `monthly` の場合、`day_of_month`（1〜31）、`start_month`、`end_month` を指定
- `day_of_month` が月の日数を超える場合は月末に補正（例: 31日指定 → 2月は28日）
- `end_month` は任意（未指定の場合は無期限）

### 4.3 入出金トランザクション管理機能

| ID | 機能 | 説明 |
|----|------|------|
| TXN-001 | トランザクション一覧取得 | 月別の入出金一覧を取得 |
| TXN-002 | トランザクション詳細取得 | 指定トランザクションの詳細を取得 |
| TXN-003 | トランザクション手動登録 | ルールに紐づかない単発の入出金を登録 |
| TXN-004 | トランザクション更新 | トランザクション内容を更新 |
| TXN-005 | ステータス更新 | トランザクションのステータスを変更 |
| TXN-006 | トランザクション削除 | トランザクションを論理削除 |

**ステータス定義:**

| ステータス | 説明 |
|-----------|------|
| `scheduled` | 予定（初期状態） |
| `completed` | 完了（実施済み） |
| `carried_over` | 繰越（翌月に持ち越し） |
| `cancelled` | キャンセル |

**ステータス遷移ルール:**

```
scheduled → completed    （実施完了）
scheduled → carried_over （繰越処理）
scheduled → cancelled    （キャンセル）
carried_over → completed （繰越先で実施完了）
carried_over → cancelled （繰越先でキャンセル）
```

- `completed` および `cancelled` からの遷移は不可
- `completed` 時は `actual_date`（実施日）を必須とする

### 4.4 繰越処理機能

| ID | 機能 | 説明 |
|----|------|------|
| CARRY-001 | 繰越実行 | `scheduled` 状態のまま月末を過ぎたトランザクションを繰越 |

**ビジネスルール:**
- 元のトランザクションのステータスを `carried_over` に変更
- 翌月に新しいトランザクションを `scheduled` で生成
- 新トランザクションの `carried_over_from` に元トランザクションIDを保持
- 元データは削除しない（履歴保持）

### 4.5 バッチ処理機能

| ID | 機能 | 説明 |
|----|------|------|
| BATCH-001 | 月次データ生成 | `monthly` ルールから翌月のトランザクションを自動生成 |

**ビジネスルール:**
- 同一ルール・同一年月のトランザクションが既に存在する場合はスキップ（冪等性）
- 日次バッチとして実行し、不足分を補完
- `day_of_month` が月の日数を超える場合は月末日に補正

### 4.6 ダッシュボード機能

| ID | 機能 | 説明 |
|----|------|------|
| DASH-001 | 月次サマリー取得 | 指定月の入金合計・支払合計・差引残高を取得 |
| DASH-002 | ステータス別集計 | 指定月のステータス別件数を取得 |

## 5. 非機能要件

### 5.1 パフォーマンス
| 項目 | 要件 |
|------|------|
| APIレスポンス | 95パーセンタイルで500ms以内 |
| 同時接続数 | 100ユーザー程度を想定 |

### 5.2 セキュリティ
| 項目 | 要件 |
|------|------|
| パスワード保存 | bcrypt によるハッシュ化 |
| 認証 | JWT（access + refresh token） |
| 通信 | HTTPS 必須 |
| CORS | フロントエンドのオリジンのみ許可 |
| 入力検証 | 全APIエンドポイントでバリデーション実施 |

### 5.3 可用性
| 項目 | 要件 |
|------|------|
| 稼働率 | 99.5% 以上 |
| バックアップ | 日次でDBバックアップ |

### 5.4 保守性
| 項目 | 要件 |
|------|------|
| ログ | 構造化ログ（JSON形式） |
| テスト | ユニットテスト・結合テスト |
| ドキュメント | API仕様はOpenAPI形式で自動生成 |

## 6. 画面要件

### 6.1 ログイン画面
| 項目 | 内容 |
|------|------|
| 入力項目 | メールアドレス、パスワード |
| 操作 | ログインボタン、サインアップ画面への遷移リンク |
| バリデーション | 必須チェック、メール形式チェック |

### 6.2 サインアップ画面
| 項目 | 内容 |
|------|------|
| 入力項目 | メールアドレス、パスワード、パスワード確認 |
| 操作 | サインアップボタン、ログイン画面への遷移リンク |
| バリデーション | 必須チェック、メール形式、パスワード強度、一致チェック |

### 6.3 ダッシュボード画面
| 項目 | 内容 |
|------|------|
| 表示項目 | 当月の入金合計、支払合計、差引残高、ステータス別件数 |
| 操作 | 月切替、一覧画面への遷移 |

### 6.4 入出金一覧画面
| 項目 | 内容 |
|------|------|
| 表示項目 | 件名、金額、種別、予定日、実施日、ステータス |
| 操作 | 月切替、新規登録、詳細表示、ステータス変更 |
| フィルタ | 種別（income/expense）、ステータス |

### 6.5 入出金登録・編集画面
| 項目 | 内容 |
|------|------|
| 入力項目 | 件名、金額、種別、予定日、メモ |
| 操作 | 保存、キャンセル |
| バリデーション | 必須チェック、金額は正の整数 |

### 6.6 ルール設定画面
| 項目 | 内容 |
|------|------|
| 表示項目 | ルール一覧（件名、種別、繰り返し種別、日） |
| 操作 | 新規登録、編集、削除 |

### 6.7 ルール登録・編集画面
| 項目 | 内容 |
|------|------|
| 入力項目 | 件名、金額、種別、繰り返し種別、日（毎月の場合）、開始月、終了月 |
| 操作 | 保存、キャンセル |
| バリデーション | 必須チェック、日は1〜31、開始月 ≤ 終了月 |
